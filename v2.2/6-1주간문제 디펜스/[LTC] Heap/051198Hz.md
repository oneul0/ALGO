# Q2. Find K Pairs with Smallest Sums
## 접근
두개의 힙에서 가장 작은 원소로 쌍을 만들고, 그중 큰 원소는 Pop한다.

## 오답
#Input
```
nums1 = [1,2,4,5,6]
nums2 = [3,5,7,9]
k = 3
```

#Output
```
[[1,3],[1,5],[1,7]]
```

#Expected
```
[[1,3],[2,3],[1,5]]
```

이전 쌍에서 더 큰 원소여서 pop하면 다른 쌍에서 쓰지 못함
그러나 다른 쌍에서 그 원소가 등장하는 경우도 있음.

## 정답
nums.count <= 10^5 이므로 완탐 사용 불가.

nums1, nums2는 이미 오름차순 정렬이며

```
nums1 = [1,7,11]
nums2 = [2,4,6]
```

위와 같다고 할 때, 모든 합은 다음과 같다.

```
      2   4   6
1     3   5   7
7     9  11  13
11   13  15  17
```

각 행별로 합이 최소인 순서쌍을 힙에 넣고 뽑으면
최소를 구할 수 있음을 알 수 있다.

# Q3. Construct Target Array With Multiple Sums
## 접근
엥 이거 어케품?

## 정답

n = 3일 때, [1, 1, 1]에서 [9, 5, 3]으로 갈 때, 전체합으로 바꿀 인덱스를 선택하는 경우는 무수히 많다.
그러니 결과에서 처음으로 돌아가는 방향을 생각해보자.

[9, 5, 3]에서, 직전 상태는

9 = 5 + 3 + x 일 것이다.
즉, x = 9 - 5 - 3 이며, x = 1임을 알 수 있다.
따라서 [1, 5, 3] 이 된다.

5 = 3 + 1 + x 이고
x = 5 - 3 - 1 이며, x = 1임을 알 수 있다.
따라서 [1, 1, 3] 이다.

3 = 1 + 1 + x이고
x = 3 - 1 - 1이며, x = 1임을 알 수 있다.
따라서 [1, 1, 1]이 된다.

그러나 [10000000, 1] 을 생각해보자.

10000000 = 1 + x이므로
x = 99999999 이다.

[99999999, 1]에서
99999999 = 1 + x이므로
x = 99999998이다.

위 과정을 계속 반복하게 되는데, 
1 <= target[i] <= 10^9 (10억) 이므로 1억을 초과하는 연산 횟수가 발생, 시간복잡도를 만족하지 않는다.

그러나, 위에서 어떤 수가 다른 수보다 클 때 까지는
계속 해당 수 에서 빼는것을 알 수 있다.

따라서 반복해서 빼는 것의 결과를 취하는 연산은 %와 같고
큰 수 % (나머지 수의 합) 
의 결과를 취할 수 있다.

그러나
[9, 5, 4] 일 때

9 = 5 + 4 + x
x = 9 - 5 - 4
x = 0인데, 원소는 0 이 될 수 없으므로
이 경우에는 만들 수 없다.

그러나 예외가 있는데
10000000 % 1 = 0 이지만, 
99999999, 1
99999998, 1
99999997, 1
...
2, 1
1, 1
로 가능한 경우이다.

따라서 나머지 수의 합이 1인 경우에도 성공한다.
